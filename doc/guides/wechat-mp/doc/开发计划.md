# 微信公众号（订阅号 / 服务号）接入 OpenClaw 的开发计划

## 1. 计划目标

本计划用于指导 `wechat-mp` 渠道的第一阶段实现，目标是在 OpenClaw 中打通“微信用户 -> 微信公众号 -> OpenClaw Agent -> 微信公众号回复”的最小可用闭环。

这里的“微信公众号”同时覆盖：

- 订阅号
- 服务号

但首版不会试图一次性覆盖公众号平台全部能力，而是优先完成最核心的消息接入、统一路由、文本回复、账号配置和基础运维能力。

第一阶段的完成标准，不是把公众号平台所有接口都做完，而是让 OpenClaw 可以稳定接入公众号消息会话，并为后续菜单、网页授权、二维码、模板消息、JS-SDK 等能力预留清晰扩展路径。

## 2. 接入定位与平台边界

### 2.1 渠道定位

`wechat-mp` 应作为一个新的独立渠道插件实现，负责把微信公众号的协议、鉴权、消息语义和平台限制适配到 OpenClaw 的统一渠道模型中。

建议插件 id 直接使用：

- `wechat-mp`

配置路径建议使用：

- `channels.wechat-mp`

后续如需多账号，沿用仓库现有约定：

- `channels.wechat-mp.accounts.<accountId>`

### 2.2 平台能力边界

微信公众号平台本质上有两条能力线：

1. **消息会话线**
   - 用户给公众号发消息
   - 微信把消息 / 事件推送到开发者 URL
   - 开发者可做 5 秒内被动回复
   - 或在用户互动窗口期内使用客服消息主动发送

2. **公众号内网页线**
   - OAuth 网页授权
   - JS-SDK 签名与前端能力
   - 菜单跳转 H5 / 小程序
   - 二维码引流和绑定

对 OpenClaw 来说，首版优先做消息会话线，把 H5 / JS-SDK / 授权等能力放到第二阶段。

### 2.3 订阅号与服务号差异

首版设计必须从一开始承认订阅号和服务号能力并不完全一致，因此不能把两者强行建模成同一能力集合。

建议把差异明确沉淀为 capability matrix：

| 能力 | 订阅号 | 服务号 | 首版策略 |
| --- | --- | --- | --- |
| 接收普通消息 / 事件推送 | 支持 | 支持 | P0 必做 |
| 5 秒内被动回复 | 支持 | 支持 | P0 必做 |
| 客服消息 | 需按平台实际权限校验 | 支持更完整 | P0 预留，优先被动回复 |
| 模板消息 | 受限 / 非首选 | 认证服务号主路径 | P1 |
| 自定义菜单 | 能力有限 / 版本相关 | 更完整 | P1 |
| 网页授权 OAuth | 有限制 | 认证服务号主路径 | P1 |
| JS-SDK | 可支持页面能力 | 可支持页面能力 | P1 |
| 用户管理 / 标签 / 黑名单 | 以权限校验为准 | 更完整 | P1 |
| 带参数二维码 | 支持 | 支持 | P1 |

结论是：

- **首版目标不是同时做平权支持**
- **首版应先把“订阅号 / 服务号共同具备的消息接入能力”做稳定**
- 服务号专属或更成熟的能力，放到 P1 / P2 增量实现

## 3. 实施范围

### 3.1 P0 范围

首版必须完成的能力：

1. 新增 `wechat-mp` 渠道插件
2. `channels.wechat-mp` 配置模型
3. `access_token` 获取、缓存、提前刷新、失效重试
4. 回调 GET 验证
5. 回调 POST 验签
6. 明文模式消息接入
7. 安全模式 / 兼容模式的消息解密与回包加密
8. XML / JSON 消息体解析与标准化
9. 文本消息入站
10. 基础事件入站（关注、取消关注、菜单点击、扫码）
11. 5 秒内被动文本回复
12. OpenClaw 统一路由与统一 reply 管线接入
13. `msgid` 去重
14. 基础状态与日志
15. 接入 `packages/channels`

### 3.2 P1 范围

第一阶段完成后建议补充：

1. 网页授权（`snsapi_base` / `snsapi_userinfo`）
2. JS-SDK 签名服务（`jsapi_ticket`）
3. 自定义菜单管理
4. 带参数二维码
5. 用户基础信息拉取
6. 模板消息发送与投递结果事件处理
7. 图片 / 语音等媒体消息入站
8. 多账号配置
9. 更完整的运行状态与探测

### 3.3 P2 范围

暂不纳入首版：

1. 全量媒体消息收发
2. 素材管理全能力
3. 用户标签 / 黑名单 / remark 等全量运营能力
4. 个性化菜单与精细化运营编排
5. 群发消息能力封装
6. 小程序 / 支付等平台扩展能力
7. 全自动账号绑定体系
8. 复杂 H5 平台工具链

## 4. 推荐实现路径

结合仓库现有架构文档，`wechat-mp` 应采用“完整插件”方式实现，优先参考 Feishu 风格，而不是复用 `wecom` / `wecom-app` / 其他旧渠道对象模型。

原因：

1. 微信公众号是新渠道，不存在宿主内已有成熟内核可直接包装
2. 它同时包含 webhook 接入、消息标准化、被动回复、客服消息、OAuth、JS-SDK 等明显的平台特性
3. 它有独立的鉴权、签名、加解密和 XML 语义，不适合硬塞进已有渠道结构

建议目录：

```text
extensions/wechat-mp/
├── index.ts
├── openclaw.plugin.json
├── package.json
└── src/
    ├── channel.ts
    ├── runtime.ts
    ├── types.ts
    ├── config.ts
    ├── accounts.ts
    ├── crypto.ts
    ├── token.ts
    ├── api.ts
    ├── webhook.ts
    ├── inbound.ts
    ├── events.ts
    ├── outbound.ts
    ├── send.ts
    ├── oauth.ts
    ├── js-sdk.ts
    ├── menu.ts
    ├── qrcode.ts
    ├── probe.ts
    ├── onboarding.ts
    └── state.ts
```

### 4.1 模块职责建议

- `channel.ts`：组装 `ChannelPlugin`
- `runtime.ts`：保存宿主 runtime 桥接
- `types.ts`：公众号消息 / 事件 / 配置 / target / session 类型
- `config.ts`：配置解析与 schema
- `accounts.ts`：多账号合并解析
- `crypto.ts`：GET 验签、POST 验签、AES 解密、加密回包
- `token.ts`：`access_token` / `jsapi_ticket` 缓存与刷新
- `api.ts`：公众号 HTTP API 封装
- `webhook.ts`：Webhook 路由注册与请求接入
- `inbound.ts`：普通消息 / 事件标准化、路由前校验
- `events.ts`：关注、扫码、菜单点击、模板投递结果等事件分流
- `outbound.ts`：适配宿主 outbound contract
- `send.ts`：被动回复 / 客服消息 / 模板消息直接发送
- `oauth.ts`：网页授权相关接口
- `js-sdk.ts`：`jsapi_ticket` 与签名服务
- `menu.ts`：菜单管理
- `qrcode.ts`：带参二维码
- `probe.ts`：健康探测
- `onboarding.ts`：配置向导
- `state.ts`：去重、缓存状态、最近活动时间

## 5. 关键架构设计

### 5.1 target 与 session 设计

微信公众号没有“群聊”概念，首版 session 设计应保持简单：

- 一个 `openid` 对应一个 DM 会话
- target 建议标准化为：`dm:<appid>:<openid>`
- session key 建议默认与 target 一致

如果未来引入：

- OAuth 绑定后的站内用户
- UnionID 聚合身份
- 同一开放平台多公众号联通

也应建立在现有 target / session 模型之上，而不是推翻重做。

### 5.2 入站标准化顺序

正确顺序应当是：

1. 验签
2. 解密
3. 解析 XML / JSON
4. 标准化为内部 message / event context
5. 做去重
6. 做安全 / allowFrom / pairing 判断
7. 路由到 agent
8. 记录 session 与最近路由
9. 生成回复
10. 适配被动回复 / 客服发送策略

不能在未校验、未去重、未授权前就直接写会话或落历史。

### 5.3 回复策略设计

公众号平台回复有两个明显限制：

1. **被动回复必须在 5 秒内完成**
2. **API 主动发送能力受平台窗口和权限约束**

因此首版建议采用下面的策略：

- 默认优先走 **被动文本回复**
- 被动回复只发送“最终聚合后的文本结果”
- 不做逐 token / 逐 block 流式输出
- 超长文本做受控分片
- 平台不适合实时连续输出的 skip / heartbeat / empty payload 必须吞掉

这与微信客服渠道类似：

- 不直接逐块照搬 OpenClaw 的流式回复
- 而是做公众号平台友好的聚合发送

### 5.4 被动回复与主动发送双通道

建议从第一版开始就把两种出站模式拆开：

1. **被动回复通道**
   - 用于用户发消息后的直接回包
   - 适配 5 秒内文本回复
   - 安全模式下需要加密回包

2. **主动发送通道**
   - 用于客服消息 / 模板消息 / 后续服务通知
   - 走公众号 API
   - 依赖 `access_token`

这样后续扩展模板消息、菜单跳转、事件通知时不会污染核心消息闭环。

### 5.5 明文 / 兼容 / 安全模式

首版不应只支持明文模式。

建议策略：

- 联调期可先支持明文模式跑通链路
- 交付目标必须包含安全模式
- 兼容模式视为安全模式的过渡态

需要明确：

- GET 验证使用 `signature`
- 安全模式 POST 验签使用 `msg_signature`
- 不能混用这两个签名口径
- 解密后必须校验 `appid`
- 加密回包时同样需要生成 `MsgSignature`

## 6. 分阶段任务

### 阶段 1：插件骨架与配置模型

目标：让 OpenClaw 识别 `wechat-mp` 渠道。

任务：

1. 新建 `extensions/wechat-mp/`
2. 增加 `openclaw.plugin.json`
3. 在 `index.ts` 中注册 `api.registerChannel({ plugin })`
4. 在 `channel.ts` 中定义 `meta`、`capabilities`、`config`、`configSchema`
5. 设计 `channels.wechat-mp` 顶层配置
6. 预留 `accounts.<accountId>` 多账号模型
7. 将统一渠道包 `packages/channels` 纳入 `wechat-mp`

产出：

- OpenClaw 能识别 `wechat-mp`
- `channels.wechat-mp` 配置结构合法
- 聚合包可按配置启用该渠道

### 阶段 2：鉴权、token 与 API 基础层

目标：完成公众号 API 的基础访问层。

任务：

1. 实现 `appid + appSecret -> access_token`
2. 增加 token 缓存
3. 增加提前刷新与被动刷新
4. 处理失效后重取逻辑
5. 处理 IP 白名单 / 管理员确认等错误场景日志
6. 封装基础 API 调用器
7. 预留 `jsapi_ticket` 获取能力

产出：

- 插件具备稳定调用公众号核心 API 的能力
- token 生命周期管理可用

### 阶段 3：Webhook、验签与消息解析

目标：完成 webhook 接入和消息 / 事件标准化。

任务：

1. 实现 GET 回调校验
2. 实现 POST 明文模式验签
3. 实现 POST 安全模式 `msg_signature` 验签
4. 实现 AES 解密与 `appid` 校验
5. 解析 XML / JSON 消息体
6. 标准化文本、图片、语音、位置、链接等基础消息
7. 标准化 subscribe / unsubscribe / scan / menu click / view 等事件
8. 实现 `msgid` 去重
9. 处理 5 秒响应要求

产出：

- 微信公众号回调进入 OpenClaw 后，能够稳定拿到标准化消息 / 事件

### 阶段 4：OpenClaw 入站对接

目标：把公众号消息接入 OpenClaw 的统一路由与回复系统。

任务：

1. 定义 `openid -> direct peer / target` 映射
2. 构建标准化 inbound context
3. 调用 `resolveAgentRoute`
4. 接入统一 reply dispatcher
5. 记录会话状态和最近路由
6. 处理基础 DM policy / allowFrom
7. 明确未关注 / 取消关注 / 菜单事件的路由边界

产出：

- 用户消息可以进入 Agent 处理链路
- 基础事件可作为系统事件或辅助上下文使用

### 阶段 5：公众号友好的出站与回复策略

目标：把 OpenClaw 的回复稳定回发给微信用户。

任务：

1. 实现被动文本回复
2. 支持安全模式回包加密
3. 实现超时场景的空串 / success 回包策略
4. 实现主动发送通道骨架
5. 处理超长文本分片
6. 合并回复块，避免发送噪音
7. 过滤 `empty` / `silent` / `heartbeat`
8. 为后续客服消息 / 模板消息保留发送适配层

产出：

- 用户能收到来自 OpenClaw 的正常回复
- 插件具备后续扩展主动发送的稳定边界

### 阶段 6：扩展能力与运维增强

目标：让渠道进入可调试、可维护、可扩展状态。

任务：

1. 输出账号配置状态
2. 输出 webhook 注册状态
3. 输出最近入站 / 出站时间
4. 增加基础错误日志和返回码分类
5. 补齐 onboarding 与配置样例
6. 增加 OAuth 能力骨架
7. 增加 JS-SDK 签名能力骨架
8. 增加菜单、二维码、模板消息封装骨架

产出：

- 具备最基本的排障能力
- 为 P1 功能扩展准备好模块边界

## 7. 首版配置建议

首版建议至少支持以下配置：

```json
{
  "channels": {
    "wechat-mp": {
      "enabled": true,
      "webhookPath": "/wechat-mp",
      "appId": "your-app-id",
      "appSecret": "your-app-secret",
      "token": "your-token",
      "encodingAESKey": "your-43-char-encoding-aes-key",
      "messageMode": "safe",
      "replyMode": "passive",
      "allowFrom": [],
      "welcomeText": "你好，我是你的 AI 公众号助手。"
    }
  }
}
```

建议后续扩展的字段：

- `accounts`
- `defaultAccount`
- `oauth.callbackBaseUrl`
- `oauth.defaultScope`
- `jsSdk.enabled`
- `menu.enabled`
- `menu.autoSync`
- `qrcode.enabled`
- `template.enabled`
- `inboundMedia`
- `apiBaseUrl`

### 7.1 首版关键字段含义

- `appId` / `appSecret`：公众号 API 凭据
- `token`：GET / POST 验签用 token
- `encodingAESKey`：安全模式消息加解密密钥
- `webhookPath`：消息与事件推送接入地址
- `messageMode`：`plain` / `compat` / `safe`
- `replyMode`：首版建议默认 `passive`
- `welcomeText`：首次关注 / 首次进入会话时可用欢迎语

## 8. 能力拆分建议

### 8.1 首版必须稳定的消息类型

建议 P0 先完整吃透：

1. 文本消息
2. subscribe / unsubscribe 事件
3. SCAN 事件
4. CLICK / VIEW 菜单事件

原因：

- 这些类型最直接影响 Agent 会话接入
- 可以支撑后续二维码引流、菜单入口和欢迎语逻辑

### 8.2 首版建议只做识别、不做深处理的类型

建议 P0 对下列类型做到“可识别、可记录、必要时降级为文本提示”：

- image
- voice
- video
- shortvideo
- location
- link

这样可以先保证系统不丢事件，再在 P1 逐步增强媒体处理。

### 8.3 OAuth / JS-SDK 的接入边界

OAuth 和 JS-SDK 很重要，但不应阻塞消息会话主路径。

建议策略：

- `oauth.ts` / `js-sdk.ts` 在首版就预留
- 先只完成服务端接口封装和配置项设计
- 暂不要求与消息主链路耦合

否则容易把首版做成“消息 + H5 一锅炖”，增加返工风险。

## 9. 风险与注意事项

### 9.1 平台前置条件风险

最大风险不在代码，而在公众号后台配置：

- 服务器 URL 不可公网访问
- Token 配置不一致
- EncodingAESKey 填写错误
- 消息模式与服务端实现不匹配
- 接口调用服务器 IP 未加入白名单
- 授权回调域名 / JS 安全域名未配置

这些问题会直接导致：

- 接入验证失败
- token 获取失败
- 安全模式验签 / 解密失败
- OAuth / JS-SDK 联调失败

### 9.2 5 秒响应风险

公众号在 5 秒内收不到响应会断开连接并重试，总共可能重试三次。

建议：

- 首版默认只返回最终文本，不做复杂流式投递
- 对处理时间不可控的场景优先空串 / success 回包
- 避免在 webhook 请求内做重型 IO
- 将主动通知和后续扩展能力从被动回复路径中拆开

### 9.3 安全模式实现风险

安全模式是首版最大的技术风险之一：

- 容易把 `signature` 和 `msg_signature` 混淆
- 容易漏做 `appid` 校验
- 容易把回包也按明文处理
- 容易在 XML / JSON 回包格式上出错

建议：

- 明文模式先跑通链路
- 再补安全模式
- 调试期充分利用官方调试工具
- 尽量使用官方推荐的算法流程，不要自创实现口径

### 9.4 access_token 中控风险

公众号 `access_token` 有效期 2 小时，重复刷新会导致旧 token 失效。

建议：

- 统一集中在 `token.ts` 里管理
- 不要在多个模块里分散刷新
- 支持提前刷新和被动刷新
- 记录 token 相关错误码与最近刷新时间

### 9.5 订阅号 / 服务号能力差异风险

如果首版直接假设“所有能力都对订阅号和服务号一视同仁”，后面会频繁返工。

建议：

- 在 `capabilities` 层显式区分
- 在 onboarding / probe 中提示账号类型与权限边界
- 需要服务号能力的模块在调用前显式探测和报错

### 9.6 自动回复冲突风险

一旦配置了开发者服务器 URL，平台后台已有的部分自动回复、自定义菜单默认行为会发生变化或失效。

建议：

- 在文档与 onboarding 中明确告知
- 首版默认由 OpenClaw 接管会话逻辑
- 不依赖公众号后台图形界面自动回复能力

## 10. MVP 验收标准

满足以下条件即可视为首版可用：

1. OpenClaw 能识别并启动 `wechat-mp` 渠道
2. 回调 GET 校验通过
3. 明文模式 POST 能正确验签和解析消息
4. 安全模式 POST 能正确验签、解密和回包加密
5. 用户文本消息能进入对应 Agent 会话
6. Agent 回复能在 5 秒内以被动文本回复回发给用户
7. `msgid` 去重有效，重复投递不会造成重复回复
8. subscribe / unsubscribe / 菜单点击 / 扫码等基础事件可被识别
9. 渠道运行状态可观察
10. 聚合包安装场景下可按 `channels.wechat-mp.enabled` 正常启用

## 11. 建议的开发顺序

推荐按下面顺序推进，避免过早进入复杂能力：

1. 插件骨架与配置
2. `access_token` 与 API 封装
3. GET / POST 验签
4. 明文模式消息解析
5. 安全模式解密与加密回包
6. 文本消息入站标准化
7. OpenClaw 路由与 reply 对接
8. 被动文本回复
9. 基础事件处理
10. 状态、日志与 onboarding
11. 再做 OAuth / JS-SDK / 菜单 / 二维码 / 模板消息

这个顺序的目标是：

- 先把消息闭环打通
- 再补安全模式
- 最后扩展平台能力

## 12. 后续扩展路线建议

### 12.1 P1：公众号业务入口能力

完成下面几项后，`wechat-mp` 才会真正从“消息渠道”升级为“公众号平台接入”：

1. 菜单管理
2. 带参数二维码
3. OAuth 授权
4. JS-SDK 签名

这样可以让公众号菜单、扫码、H5 页面和 Agent 会话形成闭环。

### 12.2 P1：服务通知能力

对服务号尤其有价值的能力包括：

1. 模板消息发送
2. 模板消息送达结果事件处理
3. 用户基础资料增强

这样可以把 OpenClaw 的消息能力延伸到“会话外通知”。

### 12.3 P2：平台型集成扩展

如果后续希望把 `wechat-mp` 做成更完整的平台插件，还可以继续扩展：

1. H5 登录态绑定
2. 站内用户与 OpenID / UnionID 映射
3. 媒体上传下载
4. 运营工具能力
5. 平台级事件自动编排

## 13. 一句话总结

`wechat-mp` 首版应优先做成一个“以消息会话接入为核心、同时为 OAuth / JS-SDK / 菜单 / 二维码 / 模板消息预留扩展边界”的完整渠道插件，先稳定打通公众号消息闭环，再逐步向公众号平台集成扩展。
