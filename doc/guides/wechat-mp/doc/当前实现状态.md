# 微信公众号插件当前实现状态

## 1. 说明

本文用于总结当前仓库中的 `wechat-mp` 插件已经实现了哪些能力、哪些能力只是做了边界预留、哪些能力仍未实现。

这里描述的是“当前代码实际状态”，不是最初计划中的理想范围。

当前结论基于以下内容：

- `doc/guides/wechat-mp/doc/开发计划.md`
- `.workflow/archives/WFS-wechat-mp-plan/IMPL_PLAN.md`
- `extensions/wechat-mp/`
- `packages/channels/`
- `packages/shared/src/cli/`

## 2. 总体结论

当前 `wechat-mp` 已经完成首版最小消息闭环：

1. `wechat-mp` 独立插件已存在，并可被 OpenClaw / 聚合包识别
2. 公众号回调 `GET / POST` 已接入
3. `plain / safe / compat` 三种消息模式具备基础处理边界
4. 文本消息与部分基础事件可以进入 OpenClaw 路由主链路
5. 回复可通过 `passive` 被动回包或 `active` 主动发送文本
6. `activeDeliveryMode` 已支持 `split / merged`
7. setup / aggregate / install hint / release surface 已接线

也就是说：

- 基础文本对话链路：已具备
- webhook 验签与安全模式基础链路：已具备
- OpenClaw routing / session / reply 接入：已具备
- 主动发送文本骨架：已具备
- 仓库集成与发布接线：已具备

但当前仍然只是一个“首版文本渠道”，还不是完整的“公众号平台接入”。

当前仍明显缺少：

- OAuth / JS-SDK / 二维码 / 用户信息 / 模板消息等公众号平台能力
- 图片、语音、视频等完整媒体消息收发
- 多账号正式化联调与运维能力
- 更完整的状态面板、错误分类与诊断能力

另外有两个重要现状需要单独说明：

1. `welcomeText` 已进入 schema、setup 和文档，但当前代码中**没有实际发送欢迎语的运行时逻辑**
2. 开发计划里提到的“XML / JSON 消息解析”，当前代码实际上是**XML 主路径**，未看到完整 JSON 入站支持

## 3. 已实现能力

### 3.1 插件与仓库接入

已实现：

- 独立插件包：`extensions/wechat-mp/`
- 渠道 ID：`wechat-mp`
- `openclaw.plugin.json` 已声明渠道元信息与配置 schema
- `index.ts` 已注册 `registerChannel({ plugin })`
- 已注册 HTTP 路由 / HTTP handler，用于接管公众号 webhook
- 已接入聚合包：`packages/channels`
- 聚合包 `openclaw.plugin.json` 已把 `wechat-mp` 纳入 channels 列表
- 根 README 已加入微信公众号入口与使用说明
- 发布脚本 `scripts/release-all.mjs` 已把 `wechat-mp` 纳入统一发布流程

结论：

- `wechat-mp` 已不是孤立实验目录，而是已接入仓库主流程的正式渠道插件

- `webhookPath`
- `messageMode`

- `webhookPath`

- `welcomeText` 虽然可写入配置，但当前插件运行时并未实际使用它发送欢迎消息

### 3.3 鉴权、token、crypto 与 webhook ingress

已实现：

- `appid + appSecret -> access_token`
- `access_token` 缓存
- 失效后清缓存并重取
- 提前刷新定时器
- `GET` 回调校验
- 明文 `POST` 验签
- 安全模式 `POST` 的 `msg_signature` 验签
- AES 解密
- 解密后的 `appid` 校验
- 被动回复加密回包
- 轻量 XML 解析与 XML 构造

当前 webhook 行为：

- `GET`：基于 `signature` 校验
- `GET + aes`：解密 `echostr` 后返回明文结果
- `POST plain`：按明文请求处理
- `POST safe/compat`：识别 `Encrypt` 字段后按密文链路验签、解密、再进入标准化流程

说明：

- 当前实现是“XML 主路径”
- 虽然开发计划中写了“XML / JSON 消息体解析与标准化”，但当前代码没有看到完整 JSON 入站解析实现
- `compat` 当前更接近“按收到的请求体形态决定 plain / encrypted 分支”，而不是单独做一套复杂兼容层

### 3.4 入站标准化、去重与 OpenClaw 主链路对接

已实现：

- `openid -> target` 映射
- `appId + openid -> sessionKey` 映射
- target 格式支持：
  - `user:<openid>`
  - `user:<openid>@<accountId>`
  - `wechat-mp:user:<openid>`
  - `wechat-mp:user:<openid>@<accountId>`
- session key 格式：`dm:<appId>:<openid>`
- 接入 `resolveAgentRoute`
- 接入 `dispatchReplyWithBufferedBlockDispatcher`
- 记录 inbound session 和最近 route
- 入站上下文当前统一写入 `CommandAuthorized=true`，slash command 可直接透传
- `dmPolicy`
- `allowFrom`
- `msgid` / dedupeKey 去重
- 基础账号状态更新

当前真正完成标准化并可进入后续处理的入站类型：

1. 文本消息 `text`
2. 基础事件：
   - `subscribe`
   - `unsubscribe`
   - `scan`
   - `click`
   - `view`

但这几类事件不是全部都进入 Agent：

- 会进入 Agent 主链路：
  - `text`
  - `subscribe`
  - `scan`
  - `click`
- 只被识别 / 记录，不进入 Agent 主链路：
  - `unsubscribe`
  - `view`

当前未完成标准化处理的类型：

- `image`
- `voice`
- `video`
- `shortvideo`
- `location`
- `link`
- 其他公众号消息类型

说明：

- `types.ts` 里虽然已经声明了部分图片消息结构，但当前 `inbound.ts` 实际只处理 `text` 和 `event`
- 不支持的消息类型当前会返回 `success`，并更新部分状态，但不会进入 Agent 主链路
- 去重状态当前持久化在本地：`~/.openclaw/wechat-mp/data/state.json`
- 去重 TTL 当前为 10 分钟

### 3.5 回复与出站

已实现：

- 被动文本回复
- 安全模式被动回包加密
- 主动文本发送辅助函数
- 公众号客服消息 API 发送文本
- `replyMode=passive`
- `replyMode=active`
- `activeDeliveryMode=merged`
- `activeDeliveryMode=split`
- host outbound surface：`wechatMpOutbound.sendText`

当前行为边界如下：

#### passive

- 等待 reply pipeline 产出最终文本
- 如果拿到最终文本，则直接回 XML
- 在 `plain` 模式下回明文 XML
- 在 `safe / compat` 且具备密钥时回加密 XML
- `activeDeliveryMode` 在 `passive` 模式下不生效

#### active + merged

- 等 reply pipeline 完成后
- 把多个文本块合并为一条消息主动发送

#### active + split

- 每个非空 chunk / 文本块单独主动发送一条消息

当前出站能力的实际边界：

- 当前真正稳定完成的是**文本发送**
- `send.ts` 只封装了主动文本发送
- `outbound.ts` 也只对外暴露文本发送
- 没有完整的媒体发送链路
- 没有模板消息发送链路
- 没有服务通知 / 定时通知能力

另外还需要明确：

- `api.ts` 中已经有 `uploadWechatMpMedia`、`createWechatMpMenu`、`deleteWechatMpMenu` 等原始 API helper
- 但这些 helper 目前**没有接入 `wechat-mp` 主消息链路，也没有形成正式的平台能力模块**
- 因此当前不能把“菜单管理 / 媒体能力”视为已实现，只能算底层 API 尝试性预留

### 3.6 API 预留能力

当前代码里已经出现但尚未形成正式插件能力的 API helper：

- 临时媒体上传
- 自定义菜单创建
- 自定义菜单删除

这些能力当前状态应视为：

- 底层 API helper 已存在
- 但没有接入正式模块边界

- 没有围绕这些能力形成主链路测试

因此它们更适合归类为“后续能力预留”，不应视为当前已完成的平台功能。

当前 onboarding 的能力边界：

- 可显示“是否已配置”

## 4. 部分实现但还不能算完整能力

### 4.1 多账号

当前状态：代码结构已预留，但还不能算正式完成

已经有的部分：

- schema 支持 `accounts` / `defaultAccount`
- route path 收集已考虑多账号路径
- target 格式支持 `@accountId`
- account config merge 已实现
- gateway 启停逻辑支持按 account 启动

仍然缺的部分：

- setup 没有完整多账号交互流程
- 缺少多账号实施文档
- 缺少多账号专项测试
- 缺少多账号状态展示与冲突治理说明
- 当前默认使用路径仍是单账号优先

结论：

- 代码结构上是 multi-account ready
- 但当前不应把它当成“稳定完成的多账号能力”

### 4.2 主动发送能力

当前状态：已完成文本 skeleton，但还不是完整主动发送体系

已经有的部分：

- 文本主动发送
- `split / merged` 两种主动发送形态
- host outbound 文本接口

仍然缺的部分：

- 模板消息
- 会话外通知能力
- 定时任务 / scheduler
- 更完整的发送失败补偿
- 更细的时间窗口 / 权限探测
- 图片、语音、视频等主动发送

结论：

- 当前 `active` 更适合视为“文本主动发送骨架”
- 还不能算完整运营消息能力

### 4.3 配置项与文档 / 类型对齐

当前状态：存在少量“已声明但未完全打通”的情况

已发现的对齐问题：

1. `welcomeText`
   - 已出现在 schema、setup、README、配置文档中
   - 但当前运行时代码没有自动发送欢迎语逻辑

2. `activeDeliveryMode`
   - 内部 schema 与 runtime 已支持
   - README / 配置文档也已说明
   - 但 setup 当前没有交互式录入该字段

3. 对外类型对齐
   - 内部 `types.ts` 已声明 `activeDeliveryMode`
   - 但 `extensions/wechat-mp/index.ts` 对外导出的 `WechatMpConfig / WechatMpAccountConfig` 接口目前未声明该字段

结论：

- 当前文档与代码主方向是一致的
- 但仍有少量“字段已宣传、实现只完成一部分”的现象，需要后续收敛

## 7. 一句话结论

当前 `wechat-mp` 已经是一个可用的“微信公众号文本渠道首版实现”：它完成了 webhook 接入、文本消息闭环、被动回复主路径、主动文本发送 skeleton 以及仓库级接线。

但它还不是完整的“公众号平台插件”。媒体能力、OAuth / JS-SDK / 二维码 / 模板消息、多账号正式化，以及更完整的运维诊断能力，仍然需要后续实现。
